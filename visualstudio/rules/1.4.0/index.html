SonarLint for Visual Studio - version 1.4.0

  • Release date: October 28, 2015
  • Release notes
  • Go to latest version

We have started working on Visual Basic .NET support in SonarLint for Visual Studio. As a first step, we have added 20 VB.NET rules, some of them already have codefixes. Also, 7 of these rules were implemented for C# too, so there are 108 C# rules altogether in this version. Here are the main changes since version 1.3.0:

VB.NET rules:

  • Properties should not be based on arrays - Rule S2365
  • Method parameters should not be reassigned - Rule S1226
  • Indexed properties with more than one parameter should not be used - Rule S2352
  • Arrays should be initialized using the "... = {}" syntax - Rule S2351
  • "Select" statements should end with a "Case Else" clause - Rule S131
  • End statements should not be used - Rule S1147
  • Signed types should be preferred to unsigned ones - Rule S2374
  • Tabulation characters should not be used - Rule S105
  • "On Error" statements should not be used - Rule S2359

VB.NET rules with codefixes:

  • "IsNot" should be used instead of "Not ... Is ..." - Rule S2358
  • Array designators "()" should be on the type, not the variable - Rule S1197
  • Array literals should be used instead of array creation expressions - Rule S2355
  • The "&" operator should be used to concatenate strings - Rule S1645
  • Multiple variables should not be declared on the same line - Rule S1659 (for C# too)

Rules supporting both C# and VB.NET:

  • Write-only properties should not be used - Rule S2376
  • Exceptions should not be thrown from property getters - Rule S2372
  • Public methods should not have multidimensional array parameters - Rule S2368
  • Multiple variables should not be declared on the same line - Rule S1659
  • Flags enumerations zero-value members should be named "None" - Rule S2346
  • Flags enumerations should explicitly initialize all their members - Rule S2345
  • Enumeration type names should not have "Flags" or "Enum" suffixes - Rule S2344
